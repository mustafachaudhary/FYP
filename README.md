## What is this?

Online learning platforms like Moodle log millions of student interaction events — clicks, resource views, quiz attempts, forum posts — but nobody has an easy way to see *what patterns are actually hiding in there*. Raw logs show you what happened; they don't tell you that struggling students consistently skip lecture notes before attempting quizzes, or that a particular sequence of behaviours is a reliable early warning sign.

This project turns that raw event data into something actionable: a full-stack platform that mines frequent learning session sequences, flags at-risk students before it's too late, and gives instructors and admins their own tailored dashboards to act on it. Built end-to-end as a final-year project — not just a prototype, but a containerised, tested, deployable system.

---

## What it can do

- **Sequence mining** — runs PrefixSpan (via SPMF) on sessionised Moodle event logs to surface frequent learning path patterns across a course
- **At-risk prediction** — gradient boosting model trained on historical engagement data; 82% accuracy, 84% precision on at-risk detection
- **Role-based dashboards** — separate interfaces for Instructors, Department Admins, and System Administrators, each scoped to what they actually need
- **Automated ETL** — Celery workers pull, clean, and transform LMS data on a schedule (daily at 2 AM by default)
- **25+ REST API endpoints** — Flask backend with JWT auth, Redis caching, and proper error handling
- **Production-ready** — Docker + Kubernetes deployment, GitHub Actions CI/CD, Prometheus monitoring, 82% test coverage

## Getting started

### Option A — Docker (recommended)

```bash
docker compose up
# Backend: http://localhost:5000
# Frontend: http://localhost:3000
```

### Option B — Lightweight pipeline only

If you just want to run the sequence analysis pipeline without the full stack:

```bash
# Install dependencies
python3 -m pip install fastapi "uvicorn[standard]" pydantic joblib scikit-learn pandas numpy openpyxl

# Run from the sequence-studio folder
cd DS_F25/fyp-code/sequence-studio
python3 backend/run_sequence_pipeline.py
python3 -m uvicorn backend.api:app --host 127.0.0.1 --port 9510 &
python3 -m http.server 8510

# Open the dashboard
# http://127.0.0.1:8510/frontend/index.html
```

## A few things worth knowing

- `data/processed/` and `outputs/` are regenerated every time the pipeline runs and are gitignored by default — don't rely on them being committed.
- Raw input files (e.g. `data/raw/Events.xlsx`) should stay under source control if they're small and non-sensitive.
- If you change any processing steps, include the regeneration commands in your PR so others can reproduce your results.
- The full-stack deployment targets Kubernetes with 3 backend replicas, 2 frontend replicas, and Celery workers — see `PROJECT_RUN_STEPS.md` for details.

## Key design decisions

A few trade-offs made during development that are worth knowing about if you're extending this:

- **Flask over Django** — chosen for flexibility and cleaner Celery integration. The cost is more boilerplate for auth and admin features.
- **MySQL over PostgreSQL** — simpler ops and better compatibility for the target institutional environment. PostgreSQL's JSON querying is genuinely better if that matters for your use case.
- **Redux Toolkit over Context API** — three dashboards with complex shared state made the overhead worth it. For simpler apps, Context would've been enough.
